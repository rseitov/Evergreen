# Паттерны и шаблоны проектирования

**Дата:** 2026-06-21
**Статус:** действующее соглашение
**Область:** весь код продукта Self-Healing SOP

> Это обязательное чтение перед реализацией любой задачи. Каждый субагент,
> исполняющий план, следует этим паттернам. Цель — единый, предсказуемый,
> тестируемый код, который читается как написанный одним человеком.

---

## 0. Базовые принципы (над всеми остальными)

- **Clean Code (Чистый код).** Код пишется для чтения людьми.
  - Понятные, произносимые, раскрывающие намерение имена; никаких сокращений и
    «магических» чисел/строк — только именованные константы.
  - Маленькие функции, делающие одну вещь на одном уровне абстракции; ранние
    возвраты вместо вложенных `if`.
  - Без дублирования (DRY), без мёртвого и закомментированного кода.
  - Комментарии объясняют «почему», а не «что»; самодокументирующийся код
    предпочтительнее комментариев.
  - Boy Scout Rule: оставляй код чище, чем нашёл.
- **Clean Architecture (Чистая архитектура).** Зависимости направлены внутрь, к
  бизнес-правилам, а не к фреймворкам.
  - Правило зависимостей: внешние слои (HTTP, ORM, БД) зависят от внутренних
    (доменные правила), но не наоборот. Домен не знает про FastAPI и SQLAlchemy.
  - Бизнес-логика изолирована от деталей доставки (HTTP) и персистентности (ORM)
    и тестируется без них.
  - Детали (БД, веб-фреймворк, внешние API) — это плагины на периферии, заменяемые
    без изменения ядра. Конкретная реализация этого правила в нашем стеке — §1.
- **SOLID.** Пять принципов ООП-дизайна, обязательны:
  - **S — Single Responsibility:** у модуля/класса/функции одна причина для
    изменения (см. §1 «один файл — одна ответственность»).
  - **O — Open/Closed:** открыто для расширения, закрыто для изменения; новое
    поведение добавляем новым кодом, а не правкой работающего.
  - **L — Liskov Substitution:** реализация заменяема своим контрактом без
    сюрпризов для вызывающего.
  - **I — Interface Segregation:** узкие, специализированные интерфейсы вместо
    «толстых»; потребитель не зависит от того, что не использует.
  - **D — Dependency Inversion:** зависим от абстракций, а не от конкретики;
    зависимости передаём извне (в нашем стеке — через FastAPI DI, §9), а не
    создаём внутри.
- **TDD без исключений.** Сначала падающий тест, затем минимальная реализация.
  Порядок шагов: написать тест → запустить, убедиться что падает → реализовать →
  запустить, убедиться что зелёный → коммит. Дисциплина не адаптируется.
- **DRY.** Повторяющаяся логика выносится в хелпер/зависимость. Дублирование
  кода между задачами — сигнал, что нужен общий модуль.
- **YAGNI.** Не пишем код «на будущее». Если фича не в текущем плане — её нет.
  Абстракции вводим только когда есть ≥2 реальных потребителя.
- **Один файл — одна ответственность.** Если файл начал делать несколько вещей,
  его делят. Маленькие сфокусированные файлы важнее «удобства» больших.
- **Явное лучше неявного.** Никаких скрытых глобалов, неявных контекстов,
  «магии». Зависимости передаются явно через DI.

## 1. Слоистая архитектура (backend)

Чёткие границы, поток в одну сторону. Слой не знает о слоях выше.

```
routers/   — HTTP + бизнес-правила. Зависят от deps, schemas, models.
schemas/   — Pydantic-модели запросов/ответов. Не знают о БД.
deps.py    — сквозные зависимости доступа (auth, membership, роли).
models/    — SQLAlchemy-модели. Только персистентность, никакой бизнес-логики.
db.py      — engine, Session, Base. Инфраструктура.
security.py, roles.py, config.py — изолированные утилиты без зависимостей от слоёв выше.
```

**Правило направления зависимостей:** `routers → deps/schemas → models → db`.
Модель никогда не импортирует роутер. Схема никогда не импортирует модель ORM.

---

## 2. Разделение схем: Create / Update / Out

На каждую сущность — отдельные Pydantic-схемы под каждое назначение. Никогда не
переиспользуем одну схему для ввода и вывода.

```python
class ProjectCreate(BaseModel):   # тело POST — только то, что задаёт клиент
    name: str
    allowlist_domains: list[str] = []

class ProjectUpdate(BaseModel):   # тело PATCH — все поля Optional
    name: str | None = None
    allowlist_domains: list[str] | None = None

class ProjectOut(BaseModel):      # ответ — то, что отдаём наружу (включая id, created_at)
    id: str
    org_id: str
    name: str
    allowlist_domains: list[str]
    created_at: datetime
```

- `Create` — обязательные поля ввода, разумные дефолты.
- `Update` — все поля `| None = None`; применяем только переданные (`if x is not None`).
- `Out` — форма ответа; никогда не отдаём `password_hash` и внутренние поля.

## 3. Мультитенантность: изоляция по `org_id` (критично)

**Каждый** запрос к данным тенанта фильтруется по `org_id`, взятому **из пути**
и провалидированному против membership вызывающего. `org_id` из тела запроса
никогда не доверяется.

Паттерн — зависимость `get_membership`, которая читает `org_id` из path и
проверяет, что текущий пользователь — участник этой организации (иначе 404):

```python
def get_membership(
    org_id: str = Path(...),
    user: User = Depends(get_current_user),
    db: Session = Depends(get_db),
) -> Membership:
    membership = db.execute(
        select(Membership).where(
            Membership.org_id == org_id, Membership.user_id == user.id
        )
    ).scalar_one_or_none()
    if membership is None:
        raise HTTPException(status_code=404, detail="Organization not found")
    return membership
```

- Чужая организация возвращает **404**, а не 403 — не раскрываем существование
  ресурса вне тенанта.
- Все запросы внутри роутера дополнительно фильтруют по `org_id` (см. §5).
- Это и продуктовое, и юридическое требование (152-ФЗ, изоляция данных).

## 4. Контроль доступа: роли через зависимости

Иерархия фиксирована: `viewer < editor < owner`. Проверка — фабрика
зависимостей `require_role(min_role)`, а не разбросанные `if` по роутерам.

```python
@router.post("/projects", response_model=ProjectOut, status_code=201)
def create_project(
    org_id: str,
    payload: ProjectCreate,
    _m: Membership = Depends(require_role("editor")),  # гейт прямо в сигнатуре
    db: Session = Depends(get_db),
) -> Project:
    ...
```

- Чтение — `Depends(get_membership)` (любой участник).
- Запись/изменение — `Depends(require_role("editor"))`.
- Удаление/админ — `Depends(require_role("owner"))`.
- Недостаточная роль → **403**. Гейт виден в сигнатуре функции, а не спрятан внутри.

## 5. Доступ к данным: хелперы `get_<entity>_or_404`

Точечная выборка с проверкой тенанта инкапсулируется в переиспользуемый хелпер,
а не дублируется. Возвращает объект или бросает 404.

```python
def get_project_or_404(db: Session, org_id: str, project_id: str) -> Project:
    project = db.execute(
        select(Project).where(Project.id == project_id, Project.org_id == org_id)
    ).scalar_one_or_none()
    if project is None:
        raise HTTPException(status_code=404, detail="Project not found")
    return project
```

- Каждый запрос фильтрует и по `id`, **и** по `org_id`.
- Хелпер живёт в роутере-владельце сущности и переиспользуется другими роутерами
  (например, `get_guide_or_404`, `build_guide_detail`).
- Запросы — стиль SQLAlchemy 2.0: `db.execute(select(...)).scalar_one_or_none()` /
  `.scalars().all()`. Не используем legacy `db.query(...)`.

## 6. Иммутабельность и версионность

Сущности-снапшоты (`GuideVersion` и её `Step`) **неизменяемы** после создания.
«Редактирование» = создание новой версии, а не мутация существующей.

- Новая версия получает `version_number = max(существующих) + 1`.
- Старые версии и их шаги никогда не трогаются — это аудит-история регламента.
- Указатель `current_version_id` перенаправляется на новую версию.
- Создание версии инкапсулировано в один хелпер с фиксированной сигнатурой
  `_create_version(db, guide, steps, user_id, version_number)`, чтобы и первичное
  создание, и последующие версии шли одним путём (DRY).

## 7. Портируемые типы (Postgres + SQLite)

Один и тот же код моделей работает на Postgres (прод) и SQLite (тесты) без
изменений. Это даёт быстрый тест-цикл без поднятия БД.

- **Идентификаторы:** UUID4 hex-строки (`String(32)`, генерация в Python через
  `new_id()`). Никогда не нативный UUID/serial БД.
- **JSON-поля:** портируемый `sqlalchemy.types.JSON`, не Postgres-only `JSONB`.
- **Время:** `DateTime` + `datetime.utcnow` (naive UTC), хелпер `utcnow()`.
- Общие хелперы — в `app/models/_base.py` (`new_id`, `utcnow`), переиспользуются
  всеми моделями.

Тесты создают схему через `Base.metadata.create_all` на SQLite in-memory;
прод-миграции — через Alembic на Postgres. Тесты не зависят от Alembic.

## 8. Обработка ошибок

- Доменные ошибки → `HTTPException` с точным статусом и кратким `detail`
  (английский, это внутренний API).
  - 401 — нет/невалидный токен.
  - 403 — аутентифицирован, но роль не позволяет.
  - 404 — ресурс не найден ИЛИ вне тенанта (намеренно не раскрываем).
  - 409 — конфликт (дубликат email, повторное членство).
- Валидация формы запроса — на уровне Pydantic-схем (`Literal`, `EmailStr`,
  типы). Не валидируем руками то, что валидирует схема.
- Не глотаем исключения. Не возвращаем 200 с телом-ошибкой.

---

## 9. Dependency Injection (FastAPI)

- Сессия БД — только через `Depends(get_db)`. Никогда не создаём `SessionLocal()`
  внутри роутера.
- Текущий пользователь, membership, проверка роли — через зависимости из
  `deps.py`. Бизнес-логика роутера получает уже провалидированный контекст.
- В тестах подменяем зависимости через `app.dependency_overrides` (см. `conftest.py`),
  а не патчим внутренности.

---

## 10. Транзакции

- Многошаговые записи (создание org + user + membership; guide + version + steps)
  выполняются в одной транзакции: `db.add(...)` / `db.flush()` для получения id,
  затем один `db.commit()` в конце операции.
- `db.flush()` — когда нужен сгенерированный id до коммита.
- `db.refresh(obj)` — после коммита, если нужно вернуть актуальное состояние.
- Один HTTP-запрос = одна транзакция = один коммит на успешном пути.

## 11. Именование

- Модули и файлы: `snake_case.py`, имя по сущности (`drift_event.py`).
- Классы моделей/схем: `PascalCase` (`DriftEvent`, `ProjectCreate`).
- Функции, переменные, поля: `snake_case`.
- Хелперы выборки: `get_<entity>_or_404`. Сборка ответа: `build_<entity>_detail`.
- Приватные хелперы модуля: префикс `_` (`_create_version`, `_to_out`).
- Имена, переходящие между задачами (сигнатуры функций, поля схем), фиксируются в
  блоке `Interfaces` задачи и не переименовываются между задачами.

---

## 12. Тесты

- Структура `tests/` зеркалит `routers/` один-к-одному + отдельный
  `test_isolation.py` для сквозной проверки мультитенантности.
- Тест проверяет наблюдаемое поведение через HTTP (`TestClient`), а не внутренние
  вызовы. Юнит-тесты — только для чистых утилит (`security`, `roles`).
- Каждый тест самодостаточен: поднимает свои данные через публичные эндпоинты
  (signup → project → guide), не полагается на порядок и общее состояние.
- Локальные хелперы сетапа в тест-файле (`_owner`, `_guide`) — это норма, не DRY-нарушение.
- Утверждаем и happy-path, и отказы доступа (404 на чужой тенант, 403 для viewer).

---

## 13. Коммиты

- Conventional Commits: `feat(scope): ...`, `test(scope): ...`, `fix(scope): ...`,
  `chore(scope): ...`. `scope` = подсистема (`backend`, `extension`, `ai`, `web`).
- Один коммит на задачу плана (после зелёного теста), как описано в шагах плана.
- Сообщение в повелительном наклонении, по-английски, кратко.

---

## 14. Чего не делаем

- ❌ Legacy `db.query(...)` — только `select()` + `db.execute()`.
- ❌ Нативные UUID/serial БД и `JSONB` — ломают портируемость на SQLite.
- ❌ `org_id` из тела запроса как источник истины — только из path + membership.
- ❌ Бизнес-логика в моделях ORM или в схемах Pydantic.
- ❌ Ручная валидация того, что покрывает Pydantic-схема.
- ❌ Отдача внутренних полей (`password_hash`) в `*Out`-схемах.
- ❌ Реализация без предшествующего падающего теста.
